# Podcast Summarizer - product requirements

## User story

As a busy but interested person, I want transcripts of new episodes of podcast shows that I am following to be periodically downloaded and summarized, so that I can read summaries faster than I can listen to the full podcast. 


## User experience 

1. User runs the command-line tool manually. Scheduling is a later increment.
2. Tool checks each source for episodes published since its last successful check, queues newly discovered episodes, and downloads their transcripts, e.g., `All_In-20260327-Anthropic's Generational Run, OpenAI Panics, AI Moats, Meta Loses Major Lawsuits.txt`)
3. Tool generates markdown summaries from the downloaded transcripts. e.g., `All_In-20260327-Summary-Anthropic's Generational Run, OpenAI Panics, AI Moats, Meta Loses Major Lawsuits.md`)
4. Summaries are collated into a single markdown file, e.g., `20260327-summaries.md`
5. The collated summary is copied into the user's Obsidian folder.
6. The episodes' full transcripts and summaries are moved into folders, categorized by show, e.g, 'All_In/, 20VC/'
7. Command-line tool finish with summary of operations 


## Core functionality and characteristics

**Command line tool:** All user interactions are through the command line. Tool can take a flag to skip downloading and just generate summaries off transcripts in the working folder. 

**Source of transcripts:** The source is TBD. Potential candidates are podcasts.happyscribe.com, podscripts.co, or YouTube. Each has individual show and episode pages with transcripts; the tool will likely need browser automation and web-scraping logic. The first implementation supports one chosen source only, following an evaluation against the initial shows: All-In and 20VC.

**Summary generator:** Summaries are generated by OpenAI or Claude by invoking a locally installed CLI (i.e. Codex or Claude Code) instead of calling their APIs. The prompt for making the summaries is exposed so that user can tweak it (e.g., to adjust lean towards business vs. technical details)

**Show and history tracker:** A simple JSON format tracks monitored shows, each source's last successful check, and a queue of discovered episodes. Each queued episode has a stable source ID or URL and a processing status (for example: discovered, downloaded, summarized, or failed). This is the standing v1 hypothesis, to be validated during implementation.

**Instapaper:** User has an existing Instapaper account

**Obsidian delivery:** The collated Markdown file is copied to a configured folder in the user's Obsidian vault. The destination must be configurable rather than hard-coded.


## Summary output contract

Each episode summary is a Markdown file with:

- Episode metadata: show, title, publication date, source URL, and transcript source.
- Executive takeaway: summary of the main theses/questions explored and conclusions.
- Key points: the top 3-7 news, claims, and conclusions synthesized as:
  - Point 1: news, claim, or conclusion.
    - Logic chain: A (with supporting evidence) → B (with supporting evidence) → claim or conclusion (with supporting evidence) → implications X and Y.
    - Areas of notable disagreement between podcast guests: the differing claim, logic, or evidence.
- Hot takes: top 1-3 novel but unsubstantiated observations. Label each explicitly as an inference and state the discussion evidence or reasoning that prompted it.

The exact prompt and template are stored in editable local files so the user can tune the balance of business, technical, or other detail.


## Success criteria

**Reliable** Does not require user to worry about missing episodes. Transparent operations and error handling. 
**Simple and extensble** The design, implementation, and operations of the tool leans towards simplicity


## Non-goals

v1 does not need:
- web UI
- account system
- automatic scheduling
- Instapaper delivery
- fallback to alternative sources for transcripts
 
 
## Additional considerations

Tool should be built in Python

Tool needs to be able to run on MacOS, and in the future, a Linux server
