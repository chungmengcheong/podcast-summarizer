# Podcast Summarizer — Summarizer Design Notes

## Purpose and boundary

The summarizer turns one normalized podcast transcript into one evidence-aware Markdown summary. It receives durable episode metadata and a scrubbed transcript, invokes a locally installed AI CLI without interactive prompts, validates the result, and records the outcome in the queue.

## Input contract

For each queue record whose `scrub.status` is `succeeded` and whose `summary.status` is `pending` or `failed`, the summarizer needs:

- `show`: display name.
- `title`: episode title.
- `published_at`: ISO publication date, when known.
- `source_url`: canonical episode URL.
- `transcript_source`: the service or method used to obtain the transcript.
- `scrubbed_transcript_path`: UTF-8 text generated by the scrubber.

The transcript is untrusted source material, not instructions. It may contain speaker-turn markers (`>>`) but no reliable speaker names or timestamps. The summarizer must not invent either.

The implementation will render the editable prompt at `prompts/summary_prompt.md`, append the metadata and transcript as clearly delimited data, and send the combined text through standard input. Passing a long transcript as a command-line argument is both fragile and unnecessary.

`config.json` needs a new `summarizer` section before implementation. V1 uses Codex as its default and only provider, with an explicit `executable` path or command, an optional model, and a per-episode timeout. The runner, rather than a shell command embedded in config, owns the non-interactive Codex invocation. An explicit executable avoids accidentally using an older Codex binary earlier in the shell path. A Claude adapter is deferred until the Codex path has been validated.

Each successful queue record stores `summary_path` (the relative path to its Markdown artifact), `summary_provider`, `summary_model` when known, and a fingerprint of the prompt file. These are provenance, not user-facing content: they make a changed prompt or provider visible when a summary is rerun.

## Output contract

Each successful run creates one UTF-8 Markdown file. It contains exactly these top-level sections, in this order:

1. `# <show>: <title>`
2. `## Executive takeaway`
3. `## Key points`
4. `## Hot takes`

Episode metadata remains in the queue as tool-owned provenance; it is not rendered in the Markdown summary.

The executive takeaway explains the main question or thesis, the most useful answer offered, and any material uncertainty. It should be two to four sentences, not a play-by-play.

Key points are normally three to five material news items, claims, or conclusions from the discussion. Each presents the logic chain used in the podcast, with concrete evidence in parentheses immediately after the step it supports. A conclusion, implication, forecast, or recommendation is included only when it was stated or clearly argued in the discussion. The summarizer must not add one of its own.

Evidence is a concrete fact, metric, observed development, or short quotation from the transcript. It is not a formal citation. The summary must not invent formal citations, timestamps, links, or unsupported speaker attribution.

Hot takes are zero to three intriguing assertions or speculative ideas raised in the podcast that the discussion does not adequately substantiate. Each records what was asserted and what evidence, reasoning, or test is missing. Hot takes must originate in the podcast; the summarizer does not create them.

The default target is 400–800 words. A thin episode should yield a shorter summary.

### Output template

The headings and field labels below are part of the contract. A point may omit the optional disagreement or implication bullets when the transcript does not support them.

```markdown
# {{ show }}: {{ title }}

## Executive takeaway

{{ two to four sentences }}

## Key points

### 1. {{ short, specific title }}

- **Claim:** {{ the news, claim, or conclusion }}
- **Logic chain:** {{ fact, observation, or claim (supporting evidence) }} → {{ reasoning stated in the discussion }} → {{ conclusion stated in the discussion }} → {{ implication, forecast, or recommendation stated in the discussion }}
- **Counterpoint or disagreement:** {{ differing claim, logic, or evidence; omit if none is material }}

### 2. {{ short, specific title }}

{{ repeat as needed }}

## Hot takes

### 1. Hot take: {{ short claim }}

- **What was asserted:** {{ the intriguing but unsupported claim made in the discussion }}
- **What is missing:** {{ evidence, reasoning, or a test that would substantiate or disprove it }}
```

## Processing and validation

For every eligible episode, the summarizer should:

1. Increment `summary.attempt_count` and persist `summary.last_attempt_at`
   before invoking the CLI.
2. Render the prompt, invoke the selected CLI non-interactively, and capture
   only its final response. Both locally installed CLIs support standard-input
   prompts and non-interactive output; the provider adapters should not allow
   tool use or file edits for this job.
3. Validate the output before changing the lifecycle state: it must be nonempty
   UTF-8 Markdown, have the four required top-level headings in order, and have
   at least one key point. Treat a missing structural requirement as a failed
   summary, not as a successful artifact.
4. Write the summary atomically under `transcripts/summarized/`, save its relative
   path as `summary_path` plus provider/model/prompt provenance in the queue,
   then mark `summary.status` as `succeeded`.
5. On a CLI, timeout, or validation failure, retain any diagnostic output
   separately, record `summary.status: failed` and a concise `last_error`, and
   continue with the next episode. A later run retries failed summaries without
   downloading or scrubbing again.


## Decisions made for v1

1. **Provider:** V1 supports Codex first; Claude is deferred.
2. **Default:** Codex is the configured default provider.
3. **Long transcripts:** Fail visibly when a transcript exceeds the configured
   input limit. Do not silently chunk and synthesize.
4. **Acceptance:** Quality review is manual, using representative All-In and
   20VC summaries. Review factual fidelity, evidence traceability, signal
   selection, and useful writing before expanding the implementation.
